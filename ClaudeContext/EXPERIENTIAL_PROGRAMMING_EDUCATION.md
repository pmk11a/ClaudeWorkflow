## Overview

**Experiential Programming Education** adalah metodologi pembelajaran programming yang berbasis pada pengalaman nyata dalam menyelesaikan masalah dan mengembangkan fitur pada proyek Smart Accounting DAPEN-KA. Setiap troubleshooting dan development work didokumentasikan sebagai learning case untuk membangun pemahaman web programming secara praktis.

Dokumentasi ini terdiri dari **universal methodology** yang dapat diterapkan pada berbagai situasi development, plus **detailed case studies** yang menunjukkan penerapan methodology dalam skenario real-world.

## Learning Philosophy

### Experiential Learning Cycle
1. **Concrete Experience** - Menghadapi real problem dalam development aktual

3. **Abstract Conceptualization** - Extract general principles yang transferable

## Universal Problem-Solving Methodology

### üîÑ The Complete Debugging Cycle

#### Phase 1: Problem Identification
```markdown
1. **Symptom Collection**: What is the user experiencing?
2. **Impact Assessment**: How critical is this issue?
3. **Initial Hypothesis**: What might be causing this?
```

#### Phase 2: Investigation
```markdown
1. **Data Gathering**: Logs, error messages, user reports
2. **System Analysis**: Code review, configuration check
3. **Flow Tracing**: Follow data/control flow end-to-end
```

#### Phase 3: Root Cause Analysis
```markdown
1. **Technical Analysis**: What exactly is failing?
2. **Architectural Review**: Is this a design issue?
3. **Dependency Check**: Are external factors involved?
```

#### Phase 4: Solution Design
```markdown
1. **Options Evaluation**: Multiple solution approaches
2. **Impact Assessment**: What changes are required?
3. **Risk Analysis**: What could go wrong?
```

#### Phase 5: Implementation
```markdown
1. **Incremental Changes**: Small, testable modifications
2. **Validation Steps**: Verify each change works
3. **Rollback Plan**: How to undo if issues arise
```

#### Phase 6: Verification & Documentation
```markdown
1. **Complete Testing**: End-to-end user scenario validation
2. **Documentation**: Capture problem, solution, and learnings
3. **Knowledge Transfer**: Make solution repeatable
```

## üîç Pattern Recognition Guide

### Recognize These Scenarios:
- **CSS Layout Issues** ‚Üí Check flex properties, add !important if needed
- **Menu Hierarchy Problems** ‚Üí Use L0 database field, not code parsing
- **Testing Difficulties** ‚Üí Switch from integration to unit tests with mocks
- **Documentation Chaos** ‚Üí Apply real-time documentation protocol
- **Playwright Failures** ‚Üí Analyze timing vs real bug, back to TDD if needed

### Quick Pattern Identification:
```bash
# Symptoms ‚Üí Root Cause ‚Üí Solution Pattern
Manual debugging ‚Üí Missing tests ‚Üí Implement TDD
UI not rendering ‚Üí Component/styling ‚Üí Check existing patterns
API errors ‚Üí Service logic ‚Üí Unit test business logic
Performance issues ‚Üí Architecture ‚Üí Apply Clean Architecture layers
```
